Project configuration

Project configuration

Project configuraton file stores all user configurable properties related to the project.

Project.config is the main project configuration file. X-Includes are supported here, enabling splitting the configuration across multiple files if desired. Depending on usage, this project configuration file may also appear as System.config or Extension.config.

System.config is typically provided with Sage and contains default values that Sage will use in case the same property is not defined in the current project's own configuration file. The purpose of this file is to share the same default settings across multiple Sage projects compiled in the same solution.

Extension.config is generated by Sage from an extension project when it is packed into a deliverable archive, and is in fact a modified version of the extension project's own configuration file.

Configuration sections

The various project configuration elements are discussed in the following sections, organized per configuration subject. To explore the structure of the configuration file, you can explore the project configuration schema.

Project settings

The project element is the root element of the project configuration, and is itself nested within configuration element.It provides a few important category-and-internationalization-related attributes (multiCategory, defaultCategory and sharedCategory, defaultLocale) and development-related attributes (autoGlobalize and debugMode). The development attributes are typically controlled by the build script and need not be changed manually.

  • project element
    The root element of the project configuration. Depending on the type of configuration file, this element may also be called system or extension.
    Attributes
    • General
    • name
      Specifies the name of the project.
    • Category and internationalization
    • multiCategory
      Sets a value indicating whether the project uses multiple categories. Possible values are true or false.
    • defaultCategory
      Sets the default category to use in multi-category projects where no specific category has been specified. The category with the specified name has to exist.
    • sharedCategory
      Sets the name of the shared category to use in multi-category projects. The default value is 'shared'.
    • defaultLocale
      Sets the default locale to use in multi-locale projects where no specific locale has been specified. The locale with the specified name has to exist.
    • Development
    • autoGlobalize
      Controls weather Sage will automatically internationalize XML files that reference the internationalization namespace. Possible values are true or false.
    • debugMode
      Controls how the system behaves in situations where additional debugging information may be emitted. In particular, determines whether merged scripts will be output as a single script or multiple scripts, as well as how certain XSLT errors are displayed. Possible values are true or false.

Path variables

All path-related variables can be controller through the project configuration, by updating the appropriate child element of the path. Each child element is a simple text-only element that defines one distinct path (template), and the name of the element corresponds to the name of the system path. The child elements may appear in any order.

  • paths element
    Defines the path (templates) to various files and resources.
    ~/assets {{assetpath}}/views/ {{assetpath}}/lib/modules/ ~/extensions/ {{assetpath}}/configuration/dictionary/{{locale}}.xml ~/Sitemap.config {{assetpath}}/xslt/default.xslt {{assetpath}}/configuration/Category.config _target/ ~/_target/
    Child elements
    • AssetPath
      Specifies the path to project assets. default: ~/assets
      For a multi-category project, resource path might look like this: ~/assets/{{category}}
    • View
      Specifies the template path to controller views. default: {{assetpath}}/views/
    • Module
      Specifies the template path to modules and their resources. default: {{assetpath}}/lib/modules/
    • Dictionary
      Specifies the template path to the phrase dictionaries files. default: {{assetpath}}/configuration/dictionary/{{locale}}.xml
    • DefaultStylesheet
      Specifies the template of the default stylesheet to use in case no other stylesheet has been specified. default: {{assetpath}}/views/xslt/default.xsl
    • CategoryConfiguration
      Specifies the template path to the category configuration path. default: {{assetpath}}/configuration/Category.config
    • GlobalizedDirectory
      Specifies the name of the directory in which the globalized resources will be generated. default: _target
    • GlobalizedDirectoryForNonFileResorces
      Specifies the name of the directory in which the globalized versions of non-file resources will be generated. default: ~/_target/
    • Extension
      Specifies the path in which to look for extension archives. default: ~/extensions/

Resource libraries

Resource libraries provide a convenient way to automate inclusion of various web resources into pages. The library can be configured to be automatically included based on the request URL. If a module or one of its dependencies uses a library (such as jquery.ui for instance), and the module is used on the current view the library will be automatically included as well. Libraries can also depend on other libraries, enabling further resource reuse and organization. Additionally, a library can also be included on a view simply by referencing it with the sage:library element. Read more.

  • library element

    Defines a reusable resource library that can be used throughout the project

    Example
    Attributes
    • name
      Specifies the name of the library.
    Child elements
    • resources/resource
      Specifies the resources this library consists of. Whenever the library is used, it's resources will be added to the page, as defined with this element.
      Example
      Attributes
      • type
        Specifies the resource type. Possible values are: script, style, document and icon. moreinfo collapsed
        More Less
        Type Description
        script Specifies a client-side JavaScript resource. For script resources location may be either head or body.
        style Specifies a client-side CSS resource. For style resources location may be either head or body.
        document Specifies a server-side XML resource. For document resources location may only be data.
        icon Specifies the website favorites icon. For icon resources location may only be head.
      • name
        Specifies the resource name. This attribute is only applicable for document type resources. A documents resource is inserted as a child element into the input document, and is assigned this value as its sage:resourceName attribute.
      • location
        Specifies the location in which to include the resource. Possible values are: head, body and data. moreinfo collapsed
        More Less
        Type Description
        head Indicates that the resource should be included within the head of the document.
        body Indicates that the resource should be included within the body of the document. The resource (a <script/> or a <style/> tag) will be included just before the closing </body> tag.
        data Indicates the the resource should be included as a child element of the request's input XML document. Applicable to document type resources.
      • path
        Specifies the resource path. Unless the path is qualified, it is considered relative to the root of the web application. You can use the kelp:// URL scheme in the resource path.

        Comma-separated list of user agent id's to which the resource library applies. For instance, ie6,ie7,chrome5,firefox7,opera10,safari4.

    • dependencies/library
      Specifies libraries that this library depends on. Whenever the library is used, before it's own resources are added to the page, the resources of the dependency library will be added too.
      Example
      Attributes
      • ref
        Specifies the name of the library this that library depends on.
    • include/path
      Defines a (part of a) URL path for which the library should automatically be included. Multiple path elements can be used to provide various paths.

      The text content of this element is evaluated as a regular expression.

      Example
      customer/registration customer/\d+
    • include/always
      Indicates that the library should always be included, with every request. If this element appears, any sibling path elements are ignored and the library is included, always.

Modules

Modules provide a way to package reusable groups of logic, presentation and resources, and use them in markup as custom tags. The system ensures that any resources belonging to a module get included on views that use them. Modules can depend on other modules, which means that if you reference module A which depends on module B then module B's resources will also be added, before adding module A's resources. Read more.

  • module element

    Defines a module.

    Example
    Attributes
    • name
      Specifies the name of the module.
    • category
      This value specifies the path to the module within the module directory. For instance, if the module directory is ~/assets/modules, and module Navigation is assigned category documentation, then the full path to that module will be ~/assets/modules/documentation/Navigation. This value can contains slashes, making it possible to nest directories as desired.
    Child elements
    • tags/tag
      Defines one or more tag names that can be used with this module. By default, the name of the tag corresponding to the module is the same as the name of the module itself. If you want to change that, or if you want to have multiple tag names refer to the same module, you can define the names with this tag
      Example
      HeaderMenu FooterMenu
    • dependencies
      Specifies libraries or modules that this module depends on. Whenever this module is used, before it's own resources are added to the page, the resources of the dependencies will be added too.
      Example
      Attributes
      • ref
        Specifies the name of the library this that library depends on.

Links

Sage provides basic link management facilities, and links need to be registered in the project configuration before they can be used throughout the system. In a nutshell, you define a dictionary of links in in the configuration file, and you then use them in your markup using dedicated XML elements and attribute, or a special syntax within the regular href attribute of a elements.

Meta views

Meta views are special views that can be applied to any request, replacing or augmenting the view originally selected. These views can be invoked by setting the query string parameter view to the name of the meta view that has been defined in the project configuration file. For instance:

http://myproject.com/catalog/chewables/gum/?view=xmlx

Additionally, if registered with the IIS through web.config, the MetaExtensionRewriter HTTP handler enables use of meta views as file extensions as well as .html as the default extension. This makes it possible to switch the meta views simply by changing the extension:

http://myproject.com/catalog/chewables/gum.html
http://myproject.com/catalog/chewables/gum.xmlx

Using custom elements

For additional flexibility, Sage allows you to add your own elements to the project configuration document's root element, as long as the elements are defined in a namespace other than the project configuration namespace.
