Localize (Document transform flow)
In the localization step, any XML documents that reference internationalization namespace are processed and generated separately for each of the project's locales.
How it works
  • When localizing a resource, Sage generates a separate localized file for each locale in the project.
  • For each locale, a separate copy of the original document will be opened, if it exists. Although unusual (it's much more common to have a single template/resource for all locales), this allows having completely separate files for specific locales.
  • By default, the localized files are saved in a subdirectory '_target' within the file's source directory (this location is configurable).
Locale fallback path

For each of the locales a resource is being localized to, Sage checks weather a separate file, specific to that locale exists. If it does, it selects it instead of the more generic, default file.

With the above configuration in place, when localizing a resource "support.html", the file names Sage will attempt to open will be, in this order:

  • support-ar.html (Argentina)
  • support-la.html (Latin America)
  • support-es.html (Spain)
  • support.html (Generic)

The first existing file will be used, and the search stops.

Automatic localization

The localization can be done either:

  • on-the-fly, each time any of the related resources change, or
  • the whole project can be pre-localized in a single batch step, and Sage then configured to not attempt any localization.

In a production environment, localization will typically be prebuilt prior to deplyment, as it speeds up responses by having pregenerated the localizations already.

On development setups, this step will happen per resource and as needed; when the source file or one of it's dependencies changes.

This behavior is controlled with the autoInternationalize attribute of the project element

The localization will happen only if the document references the internationalization namespace.
