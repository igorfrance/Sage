URI schemes

Sage implements a number of URI schemes that can be used throughout the system on various places.

kelp://
Used with script and style resources to point to a merged file, this scheme allows Sage to un-merge the resources and include them separately on development, and merged on production setups. Whether the script or style will be served merged or separate is controlled by configuration attribute project/@mergeResources in project configuration. For more info see resource merging with Kelp.
sageres://
The purpose of this URI scheme is to refer to Sage resources. There are two ways the value following sageres:// is interpreted.
  1. If there is a GetResource delegate registered for that name, the resource returned will be the result of invoking this delegate. For more info, read resource providers page.
  2. In all other cases, the path is expanded to a path within the current {{assetpath}}. So if the current asset path is ~/assets, the path configuration/navigation.xml will be expanded to ~/assets/configuration/navigation.xml and the resource will be opened from that location. Additionally, if a localized resource exists (file with locale suffix added to the name), it will be opened instead of the requested name. This enable automatic usage of locale-specific document overrides, if they exist. For more info, see fallback paths.
sageresx://

The sageresx scheme allows easy opening of assembly-embedded (compiled) resources. The syntax is sageresx://{assembly-name}/{path-within-assembly}, where assembly-name is the actual name of the assembly, with any dots replaced with slashes.

For instance, if the name of the assembly that contains the embedded resource is GumsRus.Chewy.Website.dll, and the path to the resource in that assembly is Resources/Xslt/Shopping/Cart.xslt, that resource would then be properly referenced as: sageresx://gumsrus/chewy/website/resources/xslt/shopping/cart.xslt
